# チーム開発運用ルール

## 1. 基本方針

- 変更はすべてPRで管理する。
- 仕様変更と実装変更は同一PRでドキュメント更新まで完了させる。
- AI生成コードを含む場合でも、人間レビュー承認なしでマージしない。

## 2. ブランチ運用

- `main`: 保護ブランチ（直接push禁止）
- 作業ブランチ命名例:
  - `feature/<topic>`
  - `bugfix/<topic>`
  - `docs/<topic>`

## 3. PRルール

- 必須項目:
  - 背景/目的
  - 優先度/緊急度
  - 変更概要
  - テスト結果
  - 影響範囲
  - ロールバック方針
  - 関連Issue
- 最低1名のReviewer承認が必要。
- 追加コミット時は再レビューを受ける。
- 1PR1目的を原則とし、差分が大きい場合は分割する。
- 自動整形・機械的変更は、ロジック変更と分離する。
- UI変更時はスクリーンショットまたは録画を添付する。
- 緊急度が高いPRは、本文に理由を明記し、チーム連絡チャネルで通知する。
- 詳細運用は `docs/reviewer-friendly-pr-rules.md` を参照する。

## 3.1 ラベル（タグ）運用

- PRとIssueには内容に応じたラベルを設定する。
- PRは最低1つのタイプラベル（`docs`, `ci`, `bug`, `feature`, `release`）を付与する。
- MLOps本体に影響する変更は `mlops` を併記する。
- 自動ラベルが不十分な場合は作成者が手動で補正する。
- 詳細は `docs/label-taxonomy.md` を参照する。

## 4. レビューSLA

- 原則1営業日以内に初回レビュー開始。
- 2営業日以上停滞する場合は担当を再アサインする。

## 5. 品質ゲート（CI）

- 必須チェック:
  - `ruff`
  - `mypy`
  - `pytest`（カバレッジ閾値を含む）
- いずれか失敗時はマージ不可。

## 6. リリース運用

- バージョニングはSemVer。
- リリース時の手順:
  1. `CHANGELOG.md` を更新
  2. リリースPRを作成し承認を取得
  3. `main` へマージ
  4. Git tag（例: `v1.2.3`）を発行
  5. GitHub Releaseを作成
- タグ発行可否の客観判定は `docs/release-tag-governance.md` に従う。

## 7. ドキュメント運用

- `README.md` は常時最新化する。
- 仕様変更時は `docs/specification.md` を同PRで更新する。
- ルール変更時は本書（`docs/team-rules.md`）を同PRで更新する。
- Notebook運用は `docs/notebook-policy.md` に従う。

## 8. LLM運用

- プロンプト・コンテキストは `docs/llm-context/` で管理する。
- 機密情報をプロンプトやコンテキストに含めない。
- 重要な意思決定は人間が最終承認する。
