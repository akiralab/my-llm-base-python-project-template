# ラベル（タグ）運用ルール

## 1. 目的

Issue/PRの分類を統一し、優先順位付け・検索・振り返りをしやすくする。

## 2. 標準ラベル

- `docs`: ドキュメント変更
- `ci`: CI/CD・ワークフロー・自動化変更
- `mlops`: MLOps本体（学習/推論/運用）に関する変更
- `bug`: 不具合修正・不具合報告
- `feature`: 新機能・仕様追加
- `release`: リリース準備（版上げ、タグ、CHANGELOG）
- `testing`: テスト追加・修正・検証基盤
- `dependencies`: 依存ライブラリ・ランタイム更新

## 3. 適用ルール（PR）

- 最低1つの「変更タイプ」ラベルを付ける（`docs`, `ci`, `bug`, `feature`, `release`）。
- `src/` や運用処理に影響する場合は `mlops` も付与する。
- テスト追加・変更が主目的の場合は `testing` を付与する。
- 依存更新を含む場合は `dependencies` を付与する。

## 4. 適用ルール（Issue）

- バグ報告は `bug` を必須とする。
- 機能提案は `feature` を必須とする。
- MLOps領域の課題は `mlops` を併記する。

## 5. 自動付与

- PRでは `.github/labeler.yml` に基づき、変更ファイルやブランチ名から自動ラベル付与する。
- 自動付与結果が不十分な場合は、PR作成者が手動で補正する。
