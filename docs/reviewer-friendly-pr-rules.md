# Reviewer-Friendly PR 運用ガイド

## 1. 目的

レビュー時間の短縮、見落とし防止、マージ後事故の削減を目的として、PRを「レビューしやすい単位」に整える。

参考:
- https://zenn.dev/tenchijin/articles/20250206_reviewer_friendly_pull_request_secrets

## 2. PRの粒度ルール

- 1PR1目的を原則とする（機能追加、リファクタ、フォーマット修正を混在させない）。
- 差分が大きい場合は、関心ごとごとにPRを分割する。
- 自動整形・機械的置換は、ロジック変更と分離する。
- レビュワーが追える順序（基盤 -> 本体 -> テスト -> ドキュメント）でコミットを構成する。

## 3. PR本文の必須情報

- 何を変更したか（変更の要約）
- なぜ今必要か（背景・目的）
- 優先度/緊急度
- 影響範囲（機能、利用者、運用）
- 関連Issue/設計資料リンク
- テスト結果（自動/手動）
- リスクとロールバック方針

## 4. レビュワー向け情報提供

- レビュワーが読むべきポイントを明記する。
- UI変更時はスクリーンショットまたは録画を添付する。
- 重要ログや再現手順をPR本文に残す。
- 外部依存がある変更では、確認環境や前提条件を明示する。

## 5. 安全リリース観点

- 互換性を壊す変更は段階リリースを前提にする。
- 必要に応じて Feature Flag を利用し、段階的に有効化する。
- DBスキーマ変更は、ダウンタイムを避ける手順（互換性維持 -> 切替 -> 後片付け）で計画する。
- マージ直後の障害時に戻せるよう、ロールバック手順を準備する。

## 6. コミット運用

- 意味のあるコミットメッセージ（何を/なぜ）を残す。
- ノイズが多いfixupコミットは、レビュー依頼前に整理する。
- ただしレビュー容易性が上がる場合は、段階的コミットを維持してよい。

## 7. 最低チェックリスト

- [ ] 1PR1目的を満たしている
- [ ] PR本文に背景・優先度・影響範囲・テスト結果がある
- [ ] 関連Issue/設計資料リンクがある
- [ ] ロールバック方針が記載されている
- [ ] UI変更時の証跡（画像/動画）がある
